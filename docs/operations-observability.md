# Operations observability and incident status

Veud's application process exposes a small, dependency-free operational layer
for the single-host deployment. It complements the existing PostgreSQL restore
drills, HTTPS canaries, catalog telemetry, and Sentry integration without
requiring hosted CI or a paid monitoring vendor.

## Request correlation and logs

Every HTTP response receives an `X-Request-ID` generated by Veud. Production
requests are written as one-line JSON with:

- timestamp, level, and event name;
- request ID, method, path, status, and duration; and
- an aborted-request marker.

Query strings, client addresses, and request bodies are not logged. Secret-like
error fields and URL queries are redacted. Successful asset and health-check
requests are omitted from production logs to control noise, but still contribute
to the bounded in-process counters.

The public health response also includes `X-Veud-Release` and
`X-Veud-Environment`. Set these explicitly in every runtime:

```sh
VEUD_RELEASE="$(git rev-parse HEAD)"
VEUD_ENVIRONMENT=production
```

Staging derives its release from the active immutable release directory.

## Private operations dashboard

Members with the `site-operator` role can open `/admin/operations` from the
account menu. It shows:

- release and environment;
- process uptime and memory;
- completed and in-flight requests;
- recent p50/p95 latency, status buckets, and server-error rate;
- a live database probe;
- configuration presence for error aggregation, outbound email, and the
  canonical origin; and
- the last bounded set of process errors with request IDs.

The snapshot is held only in process memory. A restart intentionally clears it,
and no user identifiers, client addresses, request bodies, or query strings are
stored. Use the request ID to correlate the dashboard with structured PM2 logs
and Sentry when configured.

The dashboard classifies readiness as:

- `healthy`: database available, error rate below 1%, and p95 below 1.5 seconds;
- `degraded`: error rate from 1% to below 5%, or p95 at least 1.5 seconds; and
- `critical`: database unavailable or error rate at least 5%.

These process-local thresholds are a fast signal, not a replacement for the
multi-route HTTPS canary or database restore verification.

## Public incidents

`/status` is public and linked from the footer. Site operators can publish and
update incidents from `/admin/operations`. Each incident identifies affected
services and severity. Every status update is appended to
`ServiceIncidentEvent`; resolving an incident does not delete its public
timeline. The status page shows active incidents and resolved incidents from the
last 30 days.

Use short, factual public updates. Do not include member data, credentials,
internal addresses, exploit detail, or unconfirmed attribution.

Recommended progression:

1. `investigating` when impact is confirmed but the cause is not;
2. `identified` when the cause and repair are known;
3. `monitoring` after the repair is deployed; and
4. `resolved` once health and backlog recovery are verified.
